## 📌 项目简介

**Kaoyan Guide** 是一个面向考研场景设计的综合型智能服务平台，采用 **Spring Boot + Vue 3** 的前后端分离架构实现，融合 **大语言模型（LLM）** 与 **RAG（检索增强生成）** 能力，为考研学生提供从“信息获取”到“学习执行”的完整支持链路。

与传统的考研信息管理系统不同，本项目不只是做静态内容展示，而是进一步把平台升级为一个：

- **可查询**：院校、专业、政策、公告等信息集中管理与展示
- **可对话**：支持 AI 智能咨询与多轮上下文交流
- **可规划**：结合学习历史生成个性化每日学习计划
- **可持续使用**：通过任务管理、模拟考试、日常反馈形成长期学习闭环

> 这是一个兼具 **信息管理系统**、**智能问答系统**、**学习辅助系统** 特征的综合型毕业设计项目。

## ✨ 项目亮点

### 1. AI 智能院校咨询
系统将大模型问答能力与本地院校/专业/政策等业务数据结合，实现更贴近考研场景的智能问答能力：

- 支持围绕院校、专业、报考、政策等内容进行咨询
- 支持多轮对话与上下文记忆
- 支持 **SSE 流式输出**
- 支持按业务模块切换提示词、上下文和工具能力
- 支持本地知识增强，提高回答相关性与可信度

### 2. AI 个性化学习规划
系统能够结合用户近期学习历史、完成情况和反馈，为用户生成每日学习计划：

- 自动生成学习建议
- 按学科拆分任务
- 支持任务状态管理
- 支持基于历史表现持续调整计划
- 实现“生成—执行—反馈—再生成”的学习闭环

### 3. 完整的考研信息服务体系
平台内置院校、专业、招生政策、专业解读、公告等多类数据资源，支持考研用户进行集中式浏览与查询：

- 院校信息
- 专业目录与专业解读
- 招生政策
- 公告资讯
- 收藏、评论、互动功能

### 4. 学生端 + 管理端双端设计
项目同时提供：

- **学生端**：信息获取、智能咨询、学习规划、练习支持
- **管理端**：数据维护、公告管理、题库管理、轮播管理、统计分析

### 5. 业务闭环完整，毕设展示效果强
项目从“数据管理”延伸到“智能辅助”，具备明显的综合型系统特征：

- 传统信息管理能力
- AI 大模型集成能力
- RAG 检索增强能力
- 结构化学习计划生成能力
- 前后端分离与双端业务设计能力

## 🏗 系统架构

本项目采用典型的 **前后端分离 + AI 服务增强** 架构：

```text
┌─────────────────────────────────────────────┐
│                 前端 Vue 3                  │
│  学生端 / 管理端 / AI 咨询页 / 学习规划页      │
└─────────────────────────────────────────────┘
                     │
                     │ HTTP / SSE
                     ▼
┌─────────────────────────────────────────────┐
│              Spring Boot 后端                │
│ Controller → Service → Mapper → DB          │
│ ChatGateway / StudyPlan / Content Modules   │
└─────────────────────────────────────────────┘
        │                    │
        │                    │
        ▼                    ▼
┌───────────────┐   ┌─────────────────────────┐
│ MySQL         │   │ Redis                   │
│ 结构化业务数据 │   │ 缓存 / 记忆 / 扩展存储     │
└───────────────┘   └─────────────────────────┘
                              │
                              ▼
                  ┌─────────────────────────┐
                  │ LLM + RAG + Prompt      │
                  │ 大模型问答 / 知识增强 / 计划生成 │
                  └─────────────────────────┘
```

### 架构设计特点

- **前后端分离**：便于前后端并行开发与独立部署
- **模块化 AI 接入**：通过统一聊天入口路由不同业务模块
- **结构化数据 + 非结构化增强结合**：传统业务数据与 RAG 协同工作
- **学生端 / 管理端分层明确**：既能提供服务，也能管理内容
- **支持流式交互**：AI 模块具备更好的实时响应体验

## 🔍 核心业务逻辑

这一部分是 README 的重点，也是项目最能体现“不是简单 CRUD”的地方。

------

### 1. AI 智能院校咨询模块

这是项目最核心的智能模块之一，其目标不是简单地“调用一个聊天接口”，而是围绕业务场景构建一套完整的咨询链路。

#### 业务目标

帮助用户围绕以下问题进行自然语言咨询：

- 某院校的基本情况如何？
- 某专业是否适合报考？
- 某政策代表什么含义？
- 报考建议如何制定？
- 某学校和某专业之间如何匹配？

#### 核心流程

```
用户输入问题
   ↓
前端发起聊天请求（message + moduleType + sessionId）
   ↓
后端统一 /chat 接口接收请求
   ↓
校验用户身份、生成/复用会话 ID
   ↓
根据 moduleType 路由到具体业务模块
   ↓
构造 Prompt / 加载历史上下文 / 启用会话记忆
   ↓
RAG 检索本地知识 + 大模型生成回答
   ↓
服务端清洗输出内容
   ↓
通过 SSE 流式返回前端
```

#### 关键设计点

##### ① 统一聊天入口

系统通过统一的 `/chat` 接口承接所有 AI 请求，避免不同业务模块各自维护一套独立聊天逻辑。这样做的优势是：

- 接口风格统一
- 易于扩展新 AI 模块
- 易于统一处理鉴权、会话、流式响应等公共能力

##### ② 模块化路由

聊天系统并不是单一“机器人”，而是根据 `moduleType` 切换到不同业务场景，例如：

- `consult_college`：院校咨询
- `learning_plan`：学习规划

这意味着系统具备从“通用问答”向“场景化 AI 服务”演进的能力。

##### ③ 会话记忆机制

系统支持多轮对话，会基于用户历史上下文继续理解当前问题。这样用户不需要每次都重复完整背景信息，交互更接近真实咨询过程。

##### ④ RAG 检索增强

系统在回答问题时，不只依赖大模型本身，还会结合本地知识和业务数据进行增强，使回答：

- 更贴近项目内已有数据
- 更符合考研场景
- 更具业务约束性
- 更适合“院校咨询”这一垂直应用

##### ⑤ 流式响应

后端通过 SSE 将模型输出逐步返回给前端，用户可以实时看到回答生成过程，从而提升交互体验。

#### 模块价值

该模块体现了本项目的智能化核心，不只是“接了个大模型”，而是做了：

- 场景路由
- 会话管理
- Prompt 资源化
- RAG 知识增强
- 流式交互输出
- 后续可扩展的工具调用能力

### 2. 智能学习规划模块

这是另一个极具代表性的模块，它的核心不是“聊天”，而是**生成结构化计划**。

#### 业务目标

根据用户近期学习情况、任务完成状态和主观反馈，为用户生成一份更合理的“今日学习计划”。

#### 核心流程

```
用户请求生成某日学习计划
   ↓
查询该日期是否已有计划
   ↓
如果已有计划则直接返回
   ↓
若没有则拉取近几天学习历史
   ↓
拼接历史任务、完成状态、用户反馈
   ↓
构造 Prompt 调用 AI
   ↓
要求模型输出结构化 JSON
   ↓
后端解析并校验结果
   ↓
拆分为主计划 + 子任务入库
   ↓
前端展示计划与任务状态
```

#### 关键设计点

##### ① 避免重复生成

系统会先判断某日期是否已经生成过计划，若存在则直接返回，确保数据一致性，也避免重复调用模型。

##### ② 引入历史学习上下文

系统不会孤立地生成某一天计划，而是会读取最近几天的学习记录，例如：

- 学习任务内容
- 完成情况
- 用户反馈
- AI 上次建议

这样生成的新计划具有连续性和针对性。

##### ③ 强约束结构化输出

学习规划模块会明确要求模型输出 JSON，而不是自由文本。输出通常包含：

- `advice`：学习建议
- `tasks`：任务列表

每个任务进一步包含：

- 学科
- 内容
- 完成状态
- 排序信息
- 其他扩展字段

##### ④ 结果清洗与异常拦截

模型输出虽然智能，但并不天然可靠。因此后端需要负责：

- 清洗 JSON
- 校验字段结构
- 标准化任务数据
- 异常格式兜底处理

这一步非常关键，它决定了 AI 输出能否真正变成“可落库、可展示、可维护”的业务数据。

##### ⑤ 主从表结构存储

系统通常会把学习计划拆成两层：

- **主计划**：保存日期、学习建议、反馈等信息
- **子任务**：保存具体任务项、科目、完成情况、排序等信息

这种设计比“整段 JSON 直接存库”更利于后续：

- 前端渲染
- 任务勾选
- 进度统计
- 历史追踪
- 二次优化

#### 模块价值

这个模块体现了 AI 在垂直业务中的另一种落地方式：
 不是只回答问题，而是直接输出**结构化业务结果**，让 AI 成为业务流程的一部分。

### 3. 内容管理与信息展示模块

除了 AI 模块外，项目还搭建了一个较完整的考研信息平台底座。

#### 管理端职责

管理端负责维护平台核心数据，包括但不限于：

- 公告
- 轮播图
- 招生政策
- 专业解读
- 地区信息
- 大学信息
- 专业目录
- 题库与练习支撑数据

#### 学生端职责

学生端负责消费这些数据，并形成实际业务使用场景：

- 浏览内容
- 检索信息
- 查看详情
- 收藏与评论
- 与 AI 服务结合进行进一步咨询

#### 设计价值

这一设计使项目形成明确的数据流闭环：

```
管理端维护结构化数据
        ↓
学生端消费和浏览数据
        ↓
AI 模块在这些数据基础上进行增强服务
        ↓
用户继续产生学习与咨询行为数据
```

也就是说，本项目中的 AI 并不是悬空存在，而是建立在完整业务系统之上的增强层。

## 🗄 数据库与业务设计特点

从整体业务设计看，本项目数据库与服务层具备较强的系统性：

### 1. 结构化数据支撑传统业务

如院校、专业、政策、公告、用户、题库等，属于典型信息管理系统的数据基础。

### 2. Redis 支撑高频与扩展型能力

Redis 可用于缓存、会话记忆、临时数据和扩展型 AI 场景存储，增强系统响应能力。

### 3. 计划主表 + 子任务表设计更合理

学习规划采用主从表设计，可以更自然地支持：

- 多任务展示
- 单项勾选
- 进度管理
- 历史对比
- AI 再生成

### 4. AI 模块与业务模块解耦

AI 逻辑通过统一聊天入口和网关进行模块化组织，不直接把大模型调用硬塞到各个 Controller 中，便于维护和扩展。

## 🚀 快速开始

### 1. 环境要求

建议准备以下运行环境：

- JDK 17
- Node.js 16+
- Maven 3.8+
- MySQL 5.7 / 8.0
- Redis 6+

------

### 2. 克隆项目

```
git clone https://github.com/HZ3568/kaoyan_guide.git
cd kaoyan_guide
```

------

### 3. 初始化数据库

执行根目录下的 SQL 脚本：

```
kaoyan_guide.sql
```

完成数据库表结构与初始数据导入。

------

### 4. 启动后端

进入后端工程目录：

```
cd kaoyan_guide_springboot
```

配置 `application.yml` 中的关键参数：

- MySQL 连接信息
- Redis 连接信息
- 大模型 API 配置
- 文件上传目录配置

启动 Spring Boot 项目。

------

### 5. 启动前端

进入前端工程目录：

```
cd kaoyan_guide_vue
npm install
npm run dev
```

根据终端输出访问前端地址即可。
