## 📌 项目简介

**Kaoyan Guide** 是一个面向考研场景设计的综合型智能服务平台，采用 **Spring Boot + Vue 3** 的前后端分离架构，深度集成 **大语言模型（LLM）** 与 **RAG（检索增强生成）** 技术，为考研学生提供从”信息获取”到”学习执行”的完整支持链路。

### 🎯 核心定位

本项目不是简单的信息管理系统，而是一个融合了以下能力的智能化平台：

- **📚 可查询**：院校、专业、政策、公告等信息集中管理与展示
- **💬 可对话**：支持 AI 智能咨询与多轮上下文交流（SSE 流式输出）
- **📝 可规划**：基于学习历史生成个性化每日学习计划（结构化 AI 输出）
- **🔄 可持续**：通过任务管理、知识库问答、日常反馈形成长期学习闭环
- **📊 可管理**：完整的管理端支持内容维护、数据统计、用户管理

### 💡 项目特色

与传统考研信息系统相比，本项目的核心创新点在于：

1. **AI 能力深度集成**：不是简单调用 API，而是构建了完整的 AI 服务架构
   - 统一聊天网关 + 模块化路由
   - 会话记忆管理（Redis）
   - RAG 知识增强（向量检索）
   - 流式响应（SSE）
   - 结构化输出解析与校验

2. **业务场景化落地**：AI 不是悬空存在，而是与业务深度融合
   - 院校咨询：结合本地院校/专业数据进行增强问答
   - 学习规划：基于历史记录生成个性化计划（JSON 结构化输出）
   - 知识库问答：支持文档上传、向量化、检索增强

3. **完整的技术栈实践**：涵盖前后端分离、数据库设计、缓存优化、AI 集成等多个技术领域
   - 前后端分离架构
   - MyBatis 持久层 + Redis 缓存层
   - JWT 鉴权 + 权限控制
   - 文件上传 + 去重处理
   - 向量数据库集成（Redis Vector）

> 这是一个兼具 **信息管理系统**、**智能问答系统**、**学习辅助系统** 特征的综合型项目，适合作为毕业设计或技术实践项目。

## ✨ 项目亮点

### 1. 🤖 AI 智能院校咨询（核心亮点）

系统将大模型问答能力与本地院校/专业/政策等业务数据深度结合，实现垂直场景的智能问答：

**技术实现：**
- ✅ **统一聊天网关**：通过 `/chat` 接口统一处理所有 AI 请求，支持模块化路由
- ✅ **会话记忆管理**：基于 Redis 实现多轮对话上下文记忆（支持 TTL 配置）
- ✅ **RAG 检索增强**：集成 Redis Vector 向量数据库，支持文档向量化与相似度检索
- ✅ **SSE 流式输出**：通过 Server-Sent Events 实现实时流式响应，提升用户体验
- ✅ **Prompt 资源化管理**：提示词模板化存储，支持按模块动态加载
- ✅ **异常处理与降级**：完善的超时处理、错误提示、根因分析

**业务价值：**
- 支持围绕院校、专业、报考、政策等内容进行自然语言咨询
- 支持多轮对话，无需重复背景信息
- 结合本地知识库，回答更准确、更贴近业务场景

---

### 2. 📝 AI 个性化学习规划（核心亮点）

系统能够基于用户近期学习历史、完成情况和主观反馈，生成结构化的每日学习计划：

**技术实现：**
- ✅ **历史上下文分析**：读取前 3 天学习记录，包括任务内容、完成状态、用户反馈
- ✅ **结构化输出解析**：强约束 AI 输出 JSON 格式，包含 `advice`（建议）和 `tasks`（任务列表）
- ✅ **数据清洗与校验**：清洗 Markdown 代码块、校验 JSON 格式、过滤空内容
- ✅ **主从表设计**：计划主表 + 任务子表，支持任务勾选、进度统计、历史追踪
- ✅ **事务边界优化**：AI 调用移出事务，避免长时间占用数据库连接
- ✅ **任务去重与顺延**：支持任务签名去重、未完成任务顺延到次日

**业务价值：**
- 自动生成学习建议，按学科拆分任务
- 支持任务状态管理（完成/未完成）
- 基于历史表现持续调整计划，形成”生成—执行—反馈—再生成”的学习闭环

**关键代码设计：**
```java
// 事务边界优化：AI 调用在非事务环境中执行
public DailyStudyPlan generatePlan(Integer userId, String feedback) {
    // 1. 查询历史记录（非事务）
    // 2. 调用 AI 生成规划（非事务，耗时操作）
    // 3. 解析和校验 JSON（非事务）
    // 4. 落库（仅此部分使用 @Transactional）
    return savePlanWithTasks(...);
}
```

### 3. 📚 知识库管理与 RAG 检索

支持文档上传、向量化、检索增强，为 AI 问答提供本地知识支持：

**技术实现：**
- ✅ **文件上传与去重**：基于文件 MD5 去重，避免重复上传
- ✅ **文档向量化**：使用 Embedding 模型将文档转换为向量（1024 维）
- ✅ **向量存储**：基于 Redis Vector 存储文档向量，支持相似度检索
- ✅ **检索增强生成**：查询时先检索相关文档，再结合 LLM 生成回答
- ✅ **知识库管理**：支持文档列表、删除、同步（Redis + MySQL 双删）

### 6. 🔧 技术难点与解决方案

#### 难点 1：学习计划任务内容丢失问题

**问题描述：**
- AI 返回的任务 `content` 字段在数据库中全部为空字符串
- 根本原因：代码中使用了错误的 JSON key `”src/main/uploads/content”` 而不是 `”content”`

**解决方案：**
- 全局搜索并修复 9 处错误的 JSON key
- 增加 3 处关键日志（AI 原始响应、解析后对象、入库前实体）
- 增加数据验证，过滤空内容任务

**详细分析：** 参见 [项目难点-学习计划任务内容丢失问题排查与解决.md](./项目难点-学习计划任务内容丢失问题排查与解决.md)

#### 难点 2：AI 生成超时问题

**问题描述：**
- 调用大模型接口时经常出现 `SocketTimeoutException`
- 超时后异常栈直接暴露给前端，用户体验差

**解决方案：**
- 配置优化：增加 `timeout: 120s`、`max-retries: 1`、`max-tokens: 2000`
- 异常处理：递归提取根因，根据异常类型返回友好提示（504/503/400/500）
- 事务优化：AI 调用移出事务，避免长时间占用数据库连接
- Prompt 优化：减少输出长度要求，控制任务数量

**详细分析：** 参见计划文件 [hashed-inventing-peach.md](C:\Users\21941\.claude\plans\hashed-inventing-peach.md)

### 架构设计特点

#### 1. 前后端分离
- **前端**：Vue 3 + Vite，独立部署，支持学生端和管理端双端
- **后端**：Spring Boot，提供 RESTful API，支持 SSE 流式响应
- **优势**：前后端并行开发，独立部署，易于扩展

#### 2. 统一聊天网关
- **设计**：通过 `/chat` 接口统一处理所有 AI 请求
- **路由**：根据 `moduleType` 参数路由到不同业务模块（院校咨询、学习规划等）
- **优势**：接口风格统一，易于扩展新 AI 模块，统一处理鉴权、会话、流式响应

#### 3. 会话记忆管理
- **存储**：基于 Redis 存储会话历史，支持 TTL 配置（默认 7 天）
- **格式**：`chat:memory:{sessionId}` → List<ChatMessage>
- **优势**：支持多轮对话，无需重复背景信息

#### 4. RAG 检索增强
- **向量化**：使用 Embedding 模型将文档转换为向量（1024 维）
- **存储**：基于 Redis Vector 存储文档向量
- **检索**：查询时先检索相关文档（Top-K 相似度），再结合 LLM 生成回答
- **优势**：回答更准确，更贴近业务场景

#### 5. 结构化数据 + 非结构化增强
- **结构化数据**：MySQL 存储院校、专业、政策、学习计划等业务数据
- **非结构化增强**：Redis Vector 存储文档向量，支持 RAG 检索
- **优势**：传统业务数据与 AI 能力协同工作

#### 6. 流式交互
- **技术**：基于 SSE（Server-Sent Events）实现流式响应
- **优势**：AI 回答实时展示，用户体验更好

### 1. AI 智能院校咨询模块

这是项目最核心的智能模块之一，其目标不是简单地“调用一个聊天接口”，而是围绕业务场景构建一套完整的咨询链路。

#### 业务目标

帮助用户围绕以下问题进行自然语言咨询：

- 某院校的基本情况如何？
- 某专业是否适合报考？
- 某政策代表什么含义？
- 报考建议如何制定？
- 某学校和某专业之间如何匹配？

#### 核心流程

```
用户输入问题
   ↓
前端发起聊天请求（message + moduleType + sessionId）
   ↓
后端统一 /chat 接口接收请求
   ↓
校验用户身份、生成/复用会话 ID
   ↓
根据 moduleType 路由到具体业务模块
   ↓
构造 Prompt / 加载历史上下文 / 启用会话记忆
   ↓
RAG 检索本地知识 + 大模型生成回答
   ↓
服务端清洗输出内容
   ↓
通过 SSE 流式返回前端
```

#### 关键设计点

##### ① 统一聊天入口

系统通过统一的 `/chat` 接口承接所有 AI 请求，避免不同业务模块各自维护一套独立聊天逻辑。这样做的优势是：

- 接口风格统一
- 易于扩展新 AI 模块
- 易于统一处理鉴权、会话、流式响应等公共能力

##### ② 模块化路由

聊天系统并不是单一“机器人”，而是根据 `moduleType` 切换到不同业务场景，例如：

- `consult_college`：院校咨询
- `learning_plan`：学习规划

这意味着系统具备从“通用问答”向“场景化 AI 服务”演进的能力。

##### ③ 会话记忆机制

系统支持多轮对话，会基于用户历史上下文继续理解当前问题。这样用户不需要每次都重复完整背景信息，交互更接近真实咨询过程。

##### ④ RAG 检索增强

系统在回答问题时，不只依赖大模型本身，还会结合本地知识和业务数据进行增强，使回答：

- 更贴近项目内已有数据
- 更符合考研场景
- 更具业务约束性
- 更适合“院校咨询”这一垂直应用

##### ⑤ 流式响应

后端通过 SSE 将模型输出逐步返回给前端，用户可以实时看到回答生成过程，从而提升交互体验。

#### 模块价值

该模块体现了本项目的智能化核心，不只是“接了个大模型”，而是做了：

- 场景路由
- 会话管理
- Prompt 资源化
- RAG 知识增强
- 流式交互输出
- 后续可扩展的工具调用能力

### 2. 智能学习规划模块

这是另一个极具代表性的模块，它的核心不是“聊天”，而是**生成结构化计划**。

#### 业务目标

根据用户近期学习情况、任务完成状态和主观反馈，为用户生成一份更合理的“今日学习计划”。

#### 核心流程

```
用户请求生成某日学习计划
   ↓
查询该日期是否已有计划
   ↓
如果已有计划则直接返回
   ↓
若没有则拉取近几天学习历史
   ↓
拼接历史任务、完成状态、用户反馈
   ↓
构造 Prompt 调用 AI
   ↓
要求模型输出结构化 JSON
   ↓
后端解析并校验结果
   ↓
拆分为主计划 + 子任务入库
   ↓
前端展示计划与任务状态
```

#### 关键设计点

##### ① 避免重复生成

系统会先判断某日期是否已经生成过计划，若存在则直接返回，确保数据一致性，也避免重复调用模型。

##### ② 引入历史学习上下文

系统不会孤立地生成某一天计划，而是会读取最近几天的学习记录，例如：

- 学习任务内容
- 完成情况
- 用户反馈
- AI 上次建议

这样生成的新计划具有连续性和针对性。

##### ③ 强约束结构化输出

学习规划模块会明确要求模型输出 JSON，而不是自由文本。输出通常包含：

- `advice`：学习建议
- `tasks`：任务列表

每个任务进一步包含：

- 学科
- 内容
- 完成状态
- 排序信息
- 其他扩展字段

##### ④ 结果清洗与异常拦截

模型输出虽然智能，但并不天然可靠。因此后端需要负责：

- 清洗 JSON
- 校验字段结构
- 标准化任务数据
- 异常格式兜底处理

这一步非常关键，它决定了 AI 输出能否真正变成“可落库、可展示、可维护”的业务数据。

##### ⑤ 主从表结构存储

系统通常会把学习计划拆成两层：

- **主计划**：保存日期、学习建议、反馈等信息
- **子任务**：保存具体任务项、科目、完成情况、排序等信息

这种设计比“整段 JSON 直接存库”更利于后续：

- 前端渲染
- 任务勾选
- 进度统计
- 历史追踪
- 二次优化

#### 模块价值

这个模块体现了 AI 在垂直业务中的另一种落地方式：
 不是只回答问题，而是直接输出**结构化业务结果**，让 AI 成为业务流程的一部分。

### 4. 配置后端

进入后端工程目录：


#### 4.1 配置 application.yml

修改 `src/main/resources/application.yml` 中的关键参数：

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/kaoyan_guide?useUnicode=true&characterEncoding=utf-8&serverTimezone=GMT%2B8
    username: root
    password: your_password  # 修改为你的 MySQL 密码

    redis:
      host: localhost
      port: 6379
      password:  # 如果 Redis 有密码，填写密码

# 大模型配置
langchain4j:
  open-ai:
    chat-model:
      base-url: https://api.openai.com/v1  # 或其他兼容 OpenAI API 的服务
      api-key: your_api_key  # 修改为你的 API Key
      model-name: gpt-4  # 或其他模型
      timeout: 120s
      max-retries: 1
      max-tokens: 2000

    streaming-chat-model:
      base-url: https://api.openai.com/v1
      api-key: your_api_key
      model-name: gpt-4

  community:
    redis:
      host: localhost
      port: 6379
      dimension: 1024  # 向量维度，根据 Embedding 模型调整

# 文件上传配置
app:
  knowledge-base:
    upload-dir: ./uploads/content  # 文件上传目录
```

### 7. 常见问题

#### Q1：Redis 连接失败
**解决方案：**
- 确保 Redis 服务已启动：`redis-server`
- 检查 Redis 端口是否被占用：`netstat -an | grep 6379`
- 检查 `application.yml` 中的 Redis 配置是否正确

#### Q2：MySQL 连接失败
**解决方案：**
- 确保 MySQL 服务已启动
- 检查数据库名称、用户名、密码是否正确
- 检查 MySQL 时区配置：`serverTimezone=GMT%2B8`

#### Q3：AI 功能无法使用
**解决方案：**
- 检查 `application.yml` 中的 `langchain4j` 配置是否正确
- 确保 API Key 有效且有足够的额度
- 检查网络连接，确保可以访问 OpenAI API（或其他 LLM 服务）
- 查看后端日志，确认是否有超时或网络错误

#### Q4：文件上传失败
**解决方案：**
- 确保 `uploads/content` 目录存在且有写入权限
- 检查 `application.yml` 中的 `upload-dir` 配置是否正确
- 检查文件大小是否超过限制（默认 10MB）

#### Q5：前端页面空白
**解决方案：**
- 检查浏览器控制台是否有错误信息
- 确保后端服务已启动
- 检查 `request.js` 中的 `baseURL` 是否正确
- 清除浏览器缓存后重试

   - 数据统计：平台数据概览、可视化统计

3. **双端设计**
   - 学生端：信息获取、智能咨询、学习规划、练习支持
   - 管理端：数据维护、用户管理、知识库管理、统计分析

- ✅ 软件工程 / 计算机类专业毕业设计
- ✅ 前后端分离项目实践
- ✅ 数据库设计与后台管理系统实践
- ✅ AI 大模型接入与场景化问答实践
- ✅ RAG 检索增强生成项目实践
- ✅ 智能学习辅助系统设计实践
- ✅ 技术栈学习与实践（Spring Boot + Vue 3 + LangChain4j）
